MySQL_基础篇

发表于2023-03-08|更新于2023-03-08|MySQL

内容包括：

  1. MySQL 架构
  2. SELECT 语句执行过程
  3. MySQL 里数据的存储结构、Compact 行格式

MySQL 架构

分为Server层 和存储引擎层 ：

  • Server 层：负责建立连接、分析和执行 SQL 语句
  • 存储引擎层：负责数据的存储与读取

Server 层包含：

  • 连接器：经过 TCP 三次握手，检查权限，与客户端建立连接
  • 查询缓存：因为鸡肋，MySQL8.0 删除了该部分。查询时先到查询缓存里查找，查找不到继续执行
  • 解析器：词法分析 + 语法分析 —> 语法树。
  • 预处理器：检查表或字段是否存在；将 “*“ 号扩展为所有列
  • 优化器：选择主键索引还是普通索引，确定下 SQL 的查询语句的执行方案
  • 执行器：开始执行，与存储引擎交互

SELECT 语句执行过程

根据 MySQL 架构的流程，其执行过程如下：

  • 连接器：建立连接、管理连接、校验用户身份
  • 查询缓存：MySQL 8.0 后没有这一流程。查询缓存中如果有查询内容，直接返回，如果没有继续执行
  • 解析 SQL：对 SQL 语句进行词法分析、语法分析，构建语法树，方便后续模块读取表、字段等信息
  • 执行 SQL：
    • 预处理阶段：检查表和字段是否存在；将 “*“ 号扩展为所有列
    • 优化阶段：优化器选择查询成本最小的方式去执行
    • 执行阶段：根据执行计划执行 SQL 语句，与存储引擎中读取数据，返回给客户端

“一行记录” 的存储结构

数据存放在哪儿？

会生成哪些文件？

新建一个表（假设表名为 t_student）后，生成三个文件：

  • db.opt：存储当前数据库的默认字符和字符校验规则
  • t_student.frm：存储「表结构」，包含表的结构定义
  • t_student.ibd：存储「表数据」，具体的记录、索引等，每一张表都存于独立的一个 .ibd 文件

.ibd 文件的结构

该文件也叫表空间文件，它由段、区、页、行组成，其结构如下：

  • 行：记录都是按「行」存放的
  • 页：InnoDB 以「页」为单位进行读写的，默认一页大小为 16KB，有多种页，比如：数据页、undo 日志页、溢出页、……
  • 区：在数据量较大时，如果页与页之间的物理位置比较随机，查找对应的页就会比较费时，所以将相邻的页安排在相邻的物理位置，就可以顺序 I/O 了，因此在分配空间时是以「区」为单位进行分配，默认一个区大小为 1MB，也就是 64 页
  • 段：表空间由一个个「段」组成，一般分为：
    • 索引段：存放 B+ 树非叶子节点的区的集合
    • 数据段：存放 B+ 树叶子节点的区的集合
    • 回滚段：存放回滚数据的区的集合

InnoDB 的行格式

所谓「行格式」，就是一条记录的存储结构。

有四种：

  • Redundant：（复古）老得不能再老了，基本不用
  • Compact：（经典）MySQL 5.1 之后默认的行格式
  • Dynamic：（现代）MySQL 5.7 之后默认的行格式
  • Compressed：和 Dynamic 差不多，都是在 Compact 格式的基础上改进了一些东西

Compact 行格式

分为两个部分：额外信息 和真实数据 。

额外信息部分

  • 变长字段长度列表：对于 varchar 这种变长字段，需要记录其长度，以便截取对应数据，如果数据里没有变长字段，该部分会被去掉

计算机在识别数据的时候不会像人一样能知道每一列对应的数据，需要我们告诉计算机，比如告诉计算机从 1 号位置开始，往后 16 位，这 16 位长度表示的是对应第一列的数据 1；从 17 号位开始，往后 3 位，这 3 位表示的是第二列的数据 2；……变长的字段其长度不是固定的，所以我们需要记录变长字段的长度。

其长度的存放顺序是逆序存放的：指针指向的是记录头信息，从头信息往左就是额外信息，往右就是真实数据，所以，读取额外信息部分的数据时，是从右往左读的，所以将第一列的变长字段的长度存在第二列变长字段的长度右面，提高 CPU Cache 的命中率，这就是逆序的原因。

  • NULL 值列表：NULL 值不会存放在真实数据里面，
