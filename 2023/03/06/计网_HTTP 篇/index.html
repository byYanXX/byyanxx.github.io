认识 HTTP

就特么你叫 HTTP 啊

  • 译名：超文本传输控制协议，所谓超文本是指文字、图片、视频等信息
  • 基于 TCP（HTTP/3 是基于 UDP）
  • 是一个双向协议
  • 目前大多使用的是 HTTP/1.1 协议

label HTTP 是两点之间基于 TCP，传输像文字、图片、视频等超文本数据的一个协议。

状态码

  • 1xx：提示信息，是一种中间状态，很少用
  • 2xx：表示服务器成功 处理了客户端的请求
    • 200 OK：一切正常
    • 204 No Content：和 200 一样，只不过没有响应 body 数据
    • 206：分块下载或断点续传，表示响应 body 的数据不是所有数据，只是一部分数据
  • 3xx：表示客户端请求的资源有所改动，需要重定向 到指定 URL
    • 301：永久重定向，请求的资源已经不存在
    • 302：临时重定向，请求的资源还在，但需要指向新的 URL 去访问
    • 304：缓存重定向，不代表跳转，表示该资源在缓存区里可以继续使用，用于缓存控制
  • 4xx：表示客户端发送的报文在服务端无法处理的错误
    • 400：笼统的错误
    • 403：服务器禁止访问资源
    • 404：请求的资源在服务器里不存在或找不到
  • 5xx：表示服务端接收到客户端的报文后，处理时发生错误
    • 500：笼统的错误
    • 501：请求的功能暂不存在，待后续开发实现
    • 502：服务器作为网关或代理时，服务器本身正常，但访问后端服务器发生错误
    • 503：服务器繁忙，稍后重试

常见字段

  • Host：客户端发送请求时，指定服务器的域名
  • Connection：客户端请求服务端使用长连接（keep-alive） 机制
  • Content-Length：服务端响应客户端的请求时，告知客户端本次数据的长度 ，为解决“粘包”问题
  • Content-Type：服务端响应客户端的请求时，告知客户端本次数据的格式 ，比如是 UTF-8、……
  • Content-Encoding：服务端响应客户端的请求时，告知客户端本次数据的压缩格式 ，比如是 gzip、……

两种 HTTP 请求方法

  • GET：请求服务端发送资源，可以是文字、图片、视频等。一般写在 URL 里，只支持 ASCII 字符。可以有 body 但一般不加 body

  • POST：根据报文 body 对指定资源做出处理

  • 安全：在 HTTP 协议里的安全指的是请求的方法不会破坏服务器上的资源

  • 幂等：多次执行相同操作，得到的结果都是相同的

从 RFC 规范定义的语义上讲，GET 是安全且幂等的，POST 不具备安全与幂等的性质。

HTTP 特性

HTTP/1.1 优缺点与性能

  • 优点：简单、灵活易扩展、跨平台
    报文简单易读，协议内容可以自定义扩充，兼容多种平台。
  • 缺点：无状态、明文传输不安全
    无状态就是不用额外的资源来记录用户的状态信息，每次请求都需要验证身份信息，比较麻烦，比较简单的就是用 Cookie 技术来标识客户端，服务器再收到申请就知道是谁了；
    至于明文，就不必多说了，抓个包看一下，你号没了。

不安全这一点可以展开说说，比如：

不安全 HTTPS 解决策略（详见下面即将要讲的 HTTPS vs HTTP）
明文不安全：内容可能被窃听，用户的个人信息泄露 信息加密，公钥、私钥
内容可能被篡改：报文内容可能被篡改 摘要算法（哈希算法）+ 数字签名
假网站：公钥可以被伪造 由 CA 颁发数字证书，给公钥配上 CA 的数字签名作为证明
  • 性能：长连接、管道网络传输（默认不开启）
    HTTP/1.0 时，每次请求都要进行一遍三次握手，反复建立、断开，开销很大，长连接就解决了这种现象；
    有了长连接，就可以管道网络传输，也就是说不必再等待服务端对第一条请求的响应后才能发第二条请求，可以连着发多条请求，但服务端因为必须按请求顺序作出相应，所以容易出现响应队头阻塞 ，这在 HTTP/1.1 是没有解决的，所以这也是大部分浏览器默认关闭管道网络传输技术的原因。

HTTP/1.1 的优化

知道了 HTTP/1.1 的缺点与性能上的不足，那么就可以针对这些不足给出一些解决策略。主要从三个方面下手：

优化目标 实现方法
尽量避免发送 HTTP 请求 缓存
需要发送 HTTP 请求时，减少发送次数 减少重定向、合并请求、延迟发送
减少服务器的 HTTP 响应数据的大小 压缩（有损压缩、无损压缩）

HTTP 的缓存技术

缓存机制

客户端将第一次请求以及响应内容保存到本地磁盘，就如一条 Map 记录，key 是 URL 请求，value 是响应内容。下次再发出请求时，先找本地磁盘里是否有相同请求记录，如果有，就可以直接拿到想要的资源，不需要去请求服务端了。

问题：如果响应资源有了变化，那原来缓存的内容就过期了 ，如何解决？
解决：服务器在发送 HTTP 响应时，会在信息头写入一个估算的过期时间 ，客户端看到这个估算的过期时间就会知道自己缓存的内容是否过期，如果过期就重新发送请求，更新缓存中的响应内容。

问题：如果估算的过期时间到了，客户端重新发送请求，但如果响应资源却没有改变，服务端还有必要重发一遍响应资源吗？
解决：当然不用重发一遍，客户端在重新发送请求时，会在请求的 Etag 头部加上第一次请求时得到的响应的头部中的摘要，这个摘要是响应资源的唯一标识 。这样，服务端收到请求后，会将本地资源的摘要和客户端发来的摘要做个对比，如果不同，就发给客户端最新的响应资源，如果摘要相同，就说明客户端的缓存里的资源尚未过期，那么服务端会返回一个不含包体的 304 Not Modified 响应，客户端收到响应后，继续使用缓存里的资源。

两种缓存实现

缓存实现 具体实现方式
强制缓存：只要浏览器判断出缓存中的响应资源没有过期，那就直接使用本地缓存 通过两个 HTTP 响应头部的字段来实现，都表示资源有效期，一个是 Expires（绝对时间），另一个是 Cahche-Control（相对时间、优先级更高）
协商缓存：服务端发回的响应是 304 时，由用户决定是继续使用本地缓存还是重新请求 可以基于时间来实现，通过请求头部的 If-Modified 和响应头部的 If-Modified-Since 字段，或者基于资源摘要来实现，通过请求头部的 If-None-Match 和响应头部的 Etag 字段

减少重定向、请求合并、延迟发送

  • 减少客户端的重定向：将重定向的工作全部交由代理服务器
  • 请求合并：一次发一个包含很多小文件的大文件包，而不是单单发一个小文件，但小文件如果有改动，那就需要重发大文件包
  • 延迟发送请求：就是 “需要啥给啥”，网页请求时，没必要把还没访问到的资源都获取到，只获取当前用户需要的响应资源即可，比如，网页向下滑动时，响应到的资源才会逐渐加载出来

压缩

压缩有两种：

  • 有损压缩：35kB 的文件可以压缩到 7kB，比如：jpg 图像压缩、……
  • 无损压缩：删除不必要的空格、换行符之类；用二进制比特序列来表示数据，比如：gzip、……
